# Deploying FangTrack to Render

This is the launch checklist. Everything in the repo is ready; these are the steps that
need your Render account and a few clicks.

## 0. Before you push
- Make sure `database/market_history.sqlite` is **not** committed if it contains data you
  don't want public in the repo — but you DO want to seed the hosted DB with it. Easiest:
  keep it in the repo for the first deploy so the hosted app starts with your market
  history + your admin collection, then rely on the persistent disk from then on.
- `.env` (real secrets) must never be committed. `.env.example` documents the vars.

## 1. Create the service (Blueprint)
1. Push this repo to GitHub.
2. Render → **New → Blueprint** → pick the repo. It reads `render.yaml` and provisions:
   - a **web service** running `gunicorn wsgi:app`,
   - a **1 GB persistent disk** mounted at `/data` (so accounts + history survive redeploys),
   - env vars incl. an auto-generated `FANGTRACK_SECRET_KEY`.
3. First deploy: the build runs `pip install` + `playwright install chromium` (needed only
   if you crawl from the web service — see §4).

## 2. Env vars (Render → service → Environment)
| Var | Value |
|---|---|
| `FANGTRACK_SECRET_KEY` | auto-generated by the blueprint (or paste your own 64-hex) |
| `FANGTRACK_HTTPS` | `1` |
| `FANGTRACK_DB_PATH` | `/data/market_history.sqlite` |
| `PYTHON_VERSION` | `3.12.7` |

## 3. Seed the database on the disk
The disk at `/data` starts empty. To seed it with your current SQLite file, either:
- **Shell in** (Render → Shell) and copy the repo's DB onto the disk:
  `cp database/market_history.sqlite /data/market_history.sqlite`, **or**
- start fresh: the app auto-creates the schema; the **first account you register becomes
  admin**. (You'd re-upload your collection via the Collection → Upload a Sheet feature.)

## 4. Database: Postgres (recommended) or SQLite-on-a-disk
The default `render.yaml` provisions **Postgres** + a **cron crawler** — the clean setup:
the web service and the daily crawl share one database. The Postgres backend lives in
`database/pg.py` and activates automatically when `DATABASE_URL` is set (the blueprint wires
it). When `DATABASE_URL` is unset, the app uses SQLite unchanged, so local dev is untouched.

**Migrate your existing SQLite data into the Render Postgres (one time):**
```
# Get the DB's external connection string from the Render dashboard, then locally:
$env:DATABASE_URL = "postgres://…@…render.com/…"
python tools/migrate_to_postgres.py database/market_history.sqlite
```
This creates the schema on Postgres and copies every table (ids + sequences preserved).

**Validate the adapter first (recommended before trusting production):**
```
docker run --rm -e POSTGRES_PASSWORD=pw -p 5432:5432 postgres:16
$env:TEST_DATABASE_URL = "postgres://postgres:pw@localhost:5432/postgres"
python tests/test_pg_live.py      # exercises inserts, lastrowid, LIKE, PRAGMA, rows
```
(The pure SQL-translation logic is already covered by `tests/test_pg_translate.py`, which
runs anywhere. `test_pg_live.py` is the end-to-end check against a real Postgres — the part
that couldn't be run in the dev sandbox.)

**Daily crawl** is handled by the `fangtrack-crawl` cron service in the blueprint
(`python scheduled_crawl.py` on the shared Postgres). Adjust its `schedule:` as desired.

### SQLite-on-a-disk alternative (simpler, smaller)
If you'd rather skip Postgres for the initial beta: comment out the `databases:` block, the
`DATABASE_URL` env vars, and the cron service in `render.yaml`; give the web service a disk
with `FANGTRACK_DB_PATH=/data/market_history.sqlite`; and either trigger crawls manually
from the **Crawler** tab or set `FANGTRACK_DAILY_CRAWL_HOUR` (needs an always-on plan).

## 5. First login
Visit the URL, click **Sign up** — the first account is the admin (sees the Crawler tab,
can trigger crawls). Everyone after is a normal user with their own isolated collection,
watchlist, and alerts.
